# SAP Knowledge Base

## Описание проекта

SAP Knowledge Base — это централизованное хранилище знаний, организованное для управления инцидентами, изменениями и шаблонами документов в SAP системе. Проект обеспечивает систематизированное хранение информации по различным модулям SAP, что облегчает поиск решений и управление документацией.

**Ключевые возможности:**
- Организованная база инцидентов по модулям SAP
- Каталог стандартных и крупных изменений
- Готовые шаблоны для создания документов
- Структурированное хранилище с легким доступом к информации

## Использование

### 1. Работа с инцидентами

Инциденты организованы по модулям SAP. Каждый модуль содержит документацию по решённым проблемам.

**Модули SAP:**
- **BASIS** — Система, основы, администрирование
- **CO** — Контроллинг (Controlling)
- **FI** — Финансы (Financial Accounting)
- **HR** — Управление персоналом (Human Resources)
- **INTEGRATION** — Интеграции между системами
- **MM** — Управление материалами (Materials Management)
- **PM** — Техническое обслуживание (Plant Maintenance)
- **PP** — Планирование производства (Production Planning)
- **QM** — Управление качеством (Quality Management)
- **SD** — Сбыт (Sales & Distribution)
- **WM** — Управление складом (Warehouse Management)

**Пример добавления инцидента:**
```bash
cd knowledge-base/incidents/FI
# Создайте новый файл с описанием инцидента
```

### 2. Работа с изменениями

#### Стандартные изменения
Стандартные изменения — это повторяющиеся, предсказуемые изменения в системе.

```bash
cd knowledge-base/changes/standard-changes/
# Выберите модуль и добавьте изменение
```

#### Крупные изменения и проекты
Крупные инициативы, новые рынки, системные обновления.

```bash
cd knowledge-base/changes/changes/
# Выберите категорию и добавьте документацию
```

### 3. Использование шаблонов

Для создания новых документов используйте готовые шаблоны:

**Для инцидентов:**
```bash
cp knowledge-base/templates/incident-template.md knowledge-base/incidents/[МОДУЛЬ]/[имя_инцидента].md
```

**Для стандартных изменений:**
```bash
cp knowledge-base/templates/standard-change-template.md knowledge-base/changes/standard-changes/[МОДУЛЬ]/[имя_изменения].md
```

**Для крупных проектов:**
```bash
cp knowledge-base/templates/change-template.md knowledge-base/changes/changes/[КАТЕГОРИЯ]/[имя_проекта].md
```

## Примеры использования

### Пример 1: Документирование инцидента
1. Перейдите в нужный модуль: `knowledge-base/incidents/FI/`
2. Скопируйте шаблон: `incident-template.md`
3. Заполните шаблон с информацией об инциденте
4. Сохраните файл с описательным названием

### Пример 2: Создание стандартного изменения
1. Перейдите в: `knowledge-base/changes/standard-changes/MM/`
2. Используйте шаблон `standard-change-template.md`
3. Документируйте процесс выполнения изменения
4. Сохраните для повторного использования

### Пример 3: Отслеживание крупного проекта
1. Выберите категорию в `knowledge-base/changes/changes/`
2. Создайте документ используя `change-template.md`
3. Обновляйте статус и результаты по мере прогресса

## Рекомендации по использованию

### Лучшие практики
1. **Понятные названия файлов** — используйте структурированные названия (например, `2024-12-05_FI_Invoice_Posting_Error.md`)
2. **Дата создания** — добавляйте дату для отслеживания истории
3. **Категоризация** — размещайте документы в правильных модулях
4. **Ссылки** — добавляйте перекрёстные ссылки между связанными документами
5. **Регулярное обновление** — поддерживайте актуальность информации

### Структура документов
Каждый документ должен содержать:
- **Заголовок** — краткое описание проблемы/изменения
- **Описание** — подробное объяснение
- **Решение/Процесс** — как решить проблему или выполнить изменение
- **Тестирование** — как проверить результат
- **Ссылки** — полезные материалы и документация

## Контрибьюция

Мы приветствуем вклад в развитие этой базы знаний! Следуйте этим шагам для добавления новой информации:

### Процесс контрибьюции

1. **Форкните репозиторий**
   ```bash
   # Создайте копию репозитория в вашем аккаунте
   ```

2. **Создайте новую ветку для вашей работы**
   ```bash
   git checkout -b feature/add-[module]-[description]
   # Пример: git checkout -b feature/add-FI-monthly-reconciliation
   ```

3. **Добавьте или отредактируйте документы**
   - Используйте существующие шаблоны
   - Следуйте рекомендациям по структуре
   - Убедитесь в корректности Markdown форматирования

4. **Выполните коммит с понятным сообщением**
   ```bash
   git commit -m "Add: FI incident - Monthly reconciliation error resolution"
   ```

5. **Отправьте изменения на GitHub**
   ```bash
   git push origin feature/add-[module]-[description]
   ```

6. **Создайте Pull Request**
   - Описание должно содержать:
     - Что было добавлено/изменено
     - Почему это важно
     - Модуль(и) SAP, к которым относится

### Рекомендации для контрибьюторов
- Проверяйте орфографию и грамматику
- Придерживайтесь структуры существующих документов
- Обновляйте оглавления если необходимо
- Добавляйте ссылки на связанные документы
- Указывайте дату добавления/обновления информации
